# A2A 协作中断

## 识别中断信号

用户可能用以下方式表达中断意图：

| 信号 | 含义 |
|------|------|
| "停"、"停止"、"stop" | 立即停止所有协作 |
| "取消"、"不用了"、"算了" | 取消当前任务 |
| "方向错了"、"不是这样" | 需要调整方向，先停再重来 |
| "先等等"、"暂停" | 暂停协作，等待进一步指示 |
| "只跟你说"、"不要叫别人" | 停止协作，切换为单 bot 对话 |

## 中断处理流程

### 1. 确认中断

收到中断信号后，立即回复用户确认：

```
收到，已停止协作。{当前进度简述}

需要我做什么调整？
```

### 2. 通知协作方（如有必要）

如果已经 @ 了其他 bot 且对方可能还在处理，使用 🔕仅通知 告知取消：

```
🔕仅通知 <at user_id="ou_xxxx">bot名字</at> 之前分配的任务已取消，无需继续处理，也无需回复。
```

### 3. 处理回传

如果中断后仍收到之前 bot 的回传结果：
- 不要再 @ 回对方
- 简要告知用户："收到了 xxx 之前的处理结果，但该任务已取消。需要我整理这部分结果吗？"

## 场景示例

### 用户中途改变需求

```
用户：让后端设计一个 REST 接口
你：（已 @ 后端）
用户：等等，改成 GraphQL 吧

你的回复：
收到，切换到 GraphQL 方案。我先通知后端取消之前的 REST 任务。

🔕仅通知 <at user_id="ou_xxxx">mac-后端</at> 之前的 REST 接口设计任务已取消，无需继续。

接下来我重新分配 GraphQL 接口设计任务。
```

### 用户要求完全停止

```
用户：算了，这个需求先不做了

你的回复：
好的，已停止。目前产品已完成需求文档，后端接口设计进行中。

如果后续要继续，随时告诉我。
```

## 注意事项

- 中断后不要继续之前的协作链
- 如果用户只是想调整方向而非完全取消，确认清楚再行动
- 汇报当前进度时要简洁，不需要完整复述所有已完成的内容
