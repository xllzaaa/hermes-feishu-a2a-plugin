# A2A 协作状态查询

## 状态推断

通过群聊消息历史推断各 bot 的状态：

| 消息特征 | 推断状态 |
|---------|---------|
| 你 @ 了 bot，bot 尚未回复 | 🔄 处理中 |
| bot @ 回你并附带结果 | ✅ 已完成 |
| 你 @ 了 bot，长时间无回复 | ⏳ 等待中 / 可能阻塞 |
| 尚未 @ 的 bot（在等前置任务） | 🔜 待分配 |

## 状态汇报模板

```
## 协作进度

**任务**：{用户原始需求}

| Bot | 子任务 | 状态 |
|-----|--------|------|
| mac-产品 | 需求文档 | ✅ 已完成 |
| mac-后端 | 接口设计 | 🔄 处理中 |
| mac-前端 | 页面开发 | 🔜 待分配（等后端完成） |
| mac-测试 | 测试用例 | 🔜 待分配 |

**当前进度**：2/4 完成
**下一步**：等待后端完成接口设计后，分配给前端
```

## 阻塞识别

当某个 bot 长时间未响应时：

1. **确认是否阻塞** — 检查消息历史，是否已经 @ 过对方
2. **尝试跟进** — 重新 @ 对方询问进度

```
<at user_id="ou_xxxx">mac-后端</at> 之前分配的接口设计任务进展如何？用户在等结果。
```

3. **汇报用户** — 如果跟进后仍无响应

```
后端暂时没有响应，可能正在处理中。要继续等待还是我先基于现有信息给一个初步方案？
```

## 主动汇报时机

不需要等用户问，以下情况可以主动汇报：

- 收到关键 bot 的回传结果时，简要告知用户进度
- 所有子任务完成时，立即汇总结果
- 发现阻塞时，及时告知用户

## 注意事项

- 状态汇报要简洁，用表格而非大段文字
- 不确定的状态用"可能"标注，不要猜测
- 如果用户只是随口问一句，给一句话回复即可，不需要完整表格
